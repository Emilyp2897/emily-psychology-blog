1. Platform Overview

Mind the Gael is an online platform for mental wellbeing and physical performance coaching, built specifically for female athletes. The site delivers four product lines:

  • Personal training plans (6-week and 12-week online programmes)
  • Mental performance plans (live, parallel architecture, branched by plan_type)
  • Blog subscription (monthly content access)

Behind the public website sit four AI-driven and automated systems: a chatbot called Saoirse, a personal training plan generator, a content delivery layer, and a Stripe-based payment layer. The platform is structurally designed to operate autonomously once Emily has validated the training-plan output (referred to in this document as the v1 to v2 transition).

The site is a digital storefront and content library combined. Visitors can read free blog content, subscribe to premium articles, or buy a training plan that is generated automatically using AI. A chatbot named Saoirse answers questions and points users to relevant articles. A human (Emily) reviews each training plan before sending it during the launch phase; eventually plans will go directly to buyers without that review step.

Four external services do the heavy lifting:

  • Anthropic Claude API generates chatbot replies, training plan teasers, and full training plans (Sonnet 4.5 for full plans, Haiku 4.5 for teasers).
  • Stripe handles every payment (founding-offer subscription, programme purchases, workshops).
  • Supabase provides customer Auth, the Plans table (with the progress JSONB column for per-exercise ticks), and the plan_week_feedback table for pilot feedback. Service-role key is used on the server to bypass RLS for admin operations.
  • Resend sends every transactional email (plan delivery, notifications to Emily, crisis-flag alerts).

Two internal stores hold state:

  • Vercel Postgres (Neon-hosted) stores intake sessions and chat feedback. This is the "pipeline" store: every intake submission, teaser content, finalize state, and Stripe session lives here.
  • Supabase Postgres stores the customer-facing surface: auth.users, Plans rows with their generated content + progress JSONB, and plan_week_feedback rows. Two databases is deliberate: the pipeline (Vercel Postgres) is treated as transient pre-purchase state, while Supabase is the durable customer record once a plan is approved.
  • Static markdown files in src/content/blog/ hold every blog post and toolkit article.

3. The Site Framework (Astro & Vercel)

The website is built using a framework called Astro. It generates fast, static HTML pages (like the home page, pricing page, blog posts) and also runs small server-side endpoints (the chatbot, the plan generator). The whole thing is hosted on Vercel, which serves the pages from data centres close to your visitors so it loads quickly.

3.1 Why Astro

Astro is a modern framework that produces almost-pure HTML for content pages (no JavaScript framework overhead). For pages that need interactivity or server-side logic (the chatbot, the intake form, the Stripe checkout), Astro runs server endpoints. This gives the best of both worlds: fast content pages and dynamic functionality where needed.

3.4 Hosting on Vercel

Every push to the main branch on GitHub triggers a Vercel deployment. Vercel:

  • Builds the Astro site (running npm run build)
  • Serves the static pages from its global CDN
  • Runs the API endpoints as serverless functions when called
  • Provides the Postgres database (via Neon integration)
  • Injects environment variables at runtime
Astro is configured for hybrid output via @astrojs/vercel adapter. Pages that need server execution declare export const prerender = false at the top of the file. API endpoints in src/pages/api/ are always server-rendered. The integration with Vercel is handled by astro.config.mjs.

4.3 How blog posts are rendered

The dynamic route src/pages/content-hub/[...slug].astro catches any URL under /content-hub/ and renders the matching markdown file using the BlogPost.astro layout.   1. Emily writes drafts in Google Drive (.docx).
  2. Drafts saved to docs/Month X - ... in the project folder.
  3. I (Claude) convert to markdown, voice-edit per Emily's rules, and place in the right collection folder.
  4. Push to GitHub. Vercel rebuilds and the post is live.

4.5 Voice rules baked into all content

  • No em-dashes or en-dashes (an AI tell; replace with periods, commas, or parentheses).
  • Hyphens in compound modifiers like "6-week" are fine; they are standard English.
  • First-person, conversational, direct address ("you" not "the athlete").
  • Plain English. No hype, no motivational filler.
  • Any non-Mind-the-Gael claim must be backed by a real, citable source.

5. The Chatbot (Saoirse)

Saoirse is the chat panel in the bottom-right of every page. When a visitor types a question, the question is sent to Anthropic's Claude AI along with a "system prompt" that tells the AI who it is and how to respond. The AI's reply is shown back to the visitor. Crisis-related messages are caught BEFORE the AI is called and replaced with safe support information.

5.3 Crisis detection

Before any AI call, the user's message is scanned against a list of crisis keywords (CRISIS_TERMS in chat.ts). The list covers suicide and self-harm vocabulary plus broader mental-health-crisis indicators ("breaking point", "in crisis", "feel hopeless", etc.). If matched:

  • The AI is NOT called. No tokens spent.
  • A fixed safe response is returned. It thanks the user, directs them to emergency services (999 UK / 112 Ireland), lists Samaritans and Pieta hotlines, points to the /resources page, and gives Emily's direct email.
  • The event is logged to Postgres for Emily to review.
Why bypass the AI for crisis cases? Even with strong prompting, AI replies have non-zero variance. For mental-health-crisis situations, deterministic safety messages are required. The keyword filter is the "always on" backstop.

5.4 System prompt highlights

The system prompt sent to Claude includes (in summary):

  • Saoirse's identity: educational guide for Mind the Gael, not Emily, not a therapist.
  • Voice rules: warm, direct, plain English, no hype, no emojis, no em-dashes.
  • Output format: no markdown (the chat panel renders plain text only).
  • Length: under 100 words for almost every reply.
  • Citation rule: any claim outside Mind the Gael articles must be attributable to a real source.
  • Safety override: if any crisis indicator slips past the keyword filter, the AI is instructed to respond with the same crisis-resource message and stop.

5.5 Prompt caching

The system prompt is sent with cache_control: { type: 'ephemeral' } so Anthropic caches it for 5 minutes. Subsequent calls within that window pay ~10% of normal input-token cost (with a slight write premium on the first call).

6. Personal Training Plans (Preview-then-Buy)

A visitor fills in a form describing their sport, goals, and equipment. The system generates a SHORT teaser preview of what their plan will look like (no actual exercises). They see the teaser on a page with watermarks and theft deterrents. If they like it, they click "Buy" and pay via Stripe. After payment is confirmed, the system generates the FULL plan and emails it to them. Emily gets a notification copy of every plan.

6.1 The four-step flow

Step 1 — Intake
  Visitor fills form on /personal-training
  Submit → POST /api/programme-intake
    ├─ Red-flag detection (crisis, eating concerns, RED-S, active injury)
    │     → If matched: NO teaser, NO purchase, route to Emily, show safe message
    └─ Generate teaser (overview only, no sessions)
       Store intake + teaser in Postgres (intake_sessions table)
       Return { previewToken, teaserContent }

Step 2 — Preview
  Browser redirects to /programme-preview?token=<previewToken>
  Page fetches teaser via GET /api/programme-preview?token=...
  Displays teaser with:
    - user-select:none CSS
    - oncontextmenu disabled
    - Ctrl+P blocked
    - Watermark with client email
    - Blur on window blur
  Two Buy buttons (6-week / 12-week) carry the token to Stripe.

Step 3 — Stripe Checkout
  Buy button → GET /api/create-checkout-session?plan=programme-6-week&token=...
  Endpoint creates a Stripe Checkout session with metadata (plan + token).
  Stripe collects payment + email.
  On success → /programme-success?session_id=...&token=...

Step 4 — Finalize
  /programme-success page calls POST /api/programme-finalize { sessionId, token }
  Endpoint:
    1. Looks up intake by token from Postgres
    2. Verifies Stripe session is paid (via Stripe API)
    3. Generates the FULL plan (Anthropic call with sport profile + exercise pool)
    4. Emails the plan to the client (HTML + plain text)
    5. Emails Emily a notification copy
    6. Marks intake_sessions row as finalized
  Page shows "Your plan is on its way".

The flip from v1 to v2 is a single environment variable change: PLAN_DESTINATION_STANDARD=client in Vercel.

The /api/programme-finalize endpoint now respects this env var directly (it previously always sent the full plan to both client and Emily). The same flip controls both the legacy /api/training-plan and the new productised flow.

v1 client communication: the holding email

In v1 mode the customer has paid via Stripe but the full plan is not sent to them. Instead:

  • Emily receives the full plan with subject [REVIEW REQUIRED] {Mental|Training} Plan for {client name} so it stands out from the autonomous-mode notification.
  • The customer receives a short holding email titled "Your {duration} {mental performance | training} plan is being reviewed". It explains payment landed, the plan is drafted and with Emily for review, and they will have it by email within 48 hours.
  • The programme-success page displays the same holding message instead of "your plan is in your inbox."

The holding email is built inline in programme-finalize.ts as sendClientHoldingEmail(). It uses the same MTG logo, signature, and lavender callout box style as the full plan email so the brand is consistent.

Specialised physical tracks always force routing to Emily regardless of the env var, by the existing rule.

6.4 Teaser vs full plan (why two AI calls)

The fake-preview approach minimises API spend on visitors who don't buy:

  • Teaser (~600 output tokens, ~£0.01 per visitor): plan overview and weekly themes only. No exercises, sets, or reps.
  • Full plan (~5,000 to 10,000 output tokens, ~£0.08 to £0.16 per buyer): generated only AFTER Stripe verifies payment.
Both AI calls use Anthropic's cache_control: { type: 'ephemeral' } on the system prompt. The full-plan system prompt is ~1500 tokens, well above the 1024-token minimum cache size, so subsequent calls within a 5-minute window pay ~10% on the cached portion. The teaser prompt is smaller and may not cache.

7. Mental Performance Plans (Live)

Mental Performance Plans are a parallel product to the physical training plans. The whole stack (intake form, teaser, Stripe, full plan, email) reuses the same infrastructure, branched on a single database column. The content backbone is the 12-month Gael Performance Toolkit (the 12 mental-skill themes used across the editorial calendar).

7.1 Architecture

The mental flow runs through the same API endpoints as the physical flow. The branch happens on a single field, plan_type, which lives both on the request payload (planType: 'physical' | 'mental') and as a column on intake_sessions. Migration 003-add-plan-type-to-intake-sessions.sql introduced this column with a default of 'physical' so legacy rows stay valid.

EndpointBranch behaviour
POST /api/programme-intakeRoutes to generateMentalTeaser() when planType === 'mental', otherwise generateTeaser(). Persists plan_type alongside the intake data.
GET /api/programme-previewReturns planType in the response so the preview page can route Buy buttons to the correct Stripe plan slug.
POST /api/programme-finalizeReads plan_type from the row, calls generateMentalFullPlan() or generateFullPlan(), then sends the same email pipeline.

7.2 Intake form

The Performance Plans page (src/pages/personal-training.astro) carries two intake modals: one for physical, one for mental. The mental modal collects different fields:

  • Sport (shared)
  • Competition level (Recreational, Club, County / Regional, Inter-county / National, International)
  • Years competing (banded: less than 1, 1-3, 3-5, 5-10, 10+)
  • Plan duration (6 weeks or 12 weeks)
  • Current confidence level (1-10)
  • Performance moments to work on (multi-select: pre-game nerves, confidence, focus, recovery from mistakes, pre-performance routines, self-talk, visualisation, motivation, team and social pressure, return from injury)
  • Current mental performance routines (free text)
  • Past peak moment (free text)
  • Past struggle moment (free text)
  • Companion physical plan summary (optional, see 7.5)
  • Goals narrative (shared, required)
  • Anything else (free text)

7.3 Teaser and full plan prompts

Two new functions in the existing files:

  • generateMentalTeaser() in src/pages/api/programme-intake.ts: ~80 lines, produces the same four-section teaser (Client Snapshot, Plan Overview, Weekly Themes, What You Unlock When You Buy) but anchored to mental-performance content. Holds back specific scripts, drills, and routines until purchase.
  • generateMentalFullPlan() in src/pages/api/programme-finalize.ts: produces the full plan. Output structure: Athlete Snapshot → Plan Overview → Week-by-Week Plan (concept explainer + daily practice table + performance moment to apply it + reflection prompt) → Routines to Build → Coach Notes → Things to Track.

The mental system prompt names the 12 toolkit themes explicitly and instructs the model to anchor weekly concepts to them. Citation rule applies in full: Eysenck et al. 2007 (Attention Control Theory), Hatzigeorgiadis et al. 2011 (self-talk), Bandura 1997 (self-efficacy), PETTLEP for imagery, Self-Determination Theory for motivation, etc. Hallucinated citations are forbidden.

7.5 Cross-plan companion linking

Both intake forms carry an optional field: "Are you also doing a Mind the Gael [Mental / Physical] Plan?" If the client pastes a short summary of their other plan, the value is stored on the intake (companionPlanSummary on ConsultationWithPlanRequest) and injected into both the teaser and full-plan prompts.

The prompt-injection block is conditional. When the field is empty, nothing is added. When it has content, the model is instructed:

  • On a mental plan: anchor each week's "performance moment to apply it" to a real session from the client's physical plan; in Routines to Build, pin pre-performance routines to specific training days.
  • On a physical plan: in session cues and Coach Notes, suggest moments to layer in refocus routines, pre-performance routines, and self-talk practice that the client is already developing in their mental plan.

The injection always carries a safety rail: "Stay within what they actually described, never invent specifics about the other plan." No database join is required; the two plans link through what the client copies into the field.

7.6 Existing sport-load fields

Both intake forms collect the player's existing weekly sport load via two selects, stored on the intake as clubTrainingsPerWeek and matchesPerWeek on ConsultationWithPlanRequest:

  • clubTrainingsPerWeek: 0 / 1 / 2 / 3 / 4+
  • matchesPerWeek: 0 / 1 / 2 / variable (championship blocks etc.)

A describeSportLoad() helper (duplicated in programme-intake.ts and programme-finalize.ts) renders these as a single line for the prompt, e.g. "2 club trainings per week + 1 match per week".

The physical plan generator receives a hard load-management rule in the full-plan prompt: total weekly sessions = club trainings + matches + plan sessions; total recoverable load is 5-6 sessions for most athletes (less when matches are present). The plan must not push the player past this, must schedule a light/recovery session day-after-match, and must NEVER prescribe a heavy lift the day before or the day after a match. Coach Notes for each week should reference how the plan sits alongside the player's stated sport load.

The mental plan generator uses the same load to anchor weekly routines: matches > 0 prioritises pre-match routines, mid-game mistake reset, and post-match decompression; matches = 0 anchors routines to training days and between-session homework. The "performance moment to apply it" each week is tied to moments the player actually has in her week, not invented matches.

7.7 Season phase field

Both intake forms carry a "When is this plan for?" select with four values:

  • pre_season — Pre-season (build a base before the season starts)
  • championship_leadup — Championship lead-up (peak and taper for matches)
  • in_season — In-season (maintenance during competition)
  • off_season — Off-season (rest, recovery, lighter work / deeper mental skill work)

Optional field; an unselected value falls through as a general plan with no phase emphasis. Stored on the intake as seasonPhase on ConsultationWithPlanRequest and read by both the teaser and full-plan prompts via a describeSeasonPhase() helper that maps the enum to a human description.

When the field has a value, the prompt receives an explicit instruction block telling the model how to shape the plan structure for each phase:

  • Physical plan: pre-season = accumulation (general strength, capacity, movement quality, progressive volume). Lead-up = intensification then taper into match-readiness across final 1-2 weeks. In-season = maintenance (lower volume, top-up work, recovery prioritised). Off-season = light, broad-based work focused on mobility and unstructured play.
  • Mental plan: pre-season = foundation routines (breathing, anchors, pre-performance routines) plus habit-setting. Lead-up = composure under pressure, sharpening focus, mistake reset, championship-week routine. In-season = quick-reset tools, self-talk for fatigue, managing pressure between fixtures. Off-season = deeper identity work, reflection, longer-term mental skill development.

The same helper is duplicated in both programme-intake.ts and programme-finalize.ts so the teaser and the full plan agree on what each phase means.

7.8 Training days and active training status

Both intake forms also capture two newer fields that drive intensity calibration:

  • trainingDays — a checkbox group of which weekdays the athlete trains with their sport (e.g. ['mon', 'wed'] for a midweek footballer). Stored as a string array on the intake, read by both prompts via describeTrainingDays(). The plan generator uses this to place S&C sessions off club nights and schedule mental-performance routines around match days.
  • activeTrainingStatus — a single select capturing whether the athlete is actively competing, training but not yet competing, building back up, or not currently training. Independent of seasonPhase (which is calendar-based). Read via describeActiveTrainingStatus(), which returns explicit calibration guidance for the model (e.g. "Currently training AND competing regularly. Plan at a level that complements an athlete already at full sport load; do NOT default to beginner work."). Designed to stop the model giving a mid-season competing athlete a beginner-style plan.

7.9 Explicit OUTPUT STRUCTURE in the plan prompts

Both physical and mental full-plan prompts now enforce a fixed section structure to remove variance and dropped sections seen in earlier outputs. Each # HEADER and each bolded subsection label below must appear verbatim:

Physical plan:

  • # HOW TO READ THIS PLAN — orientation block defining RPE, Sets × Reps, Rest, Warm-up note, Cues, Cycle consideration, Ready to progress, Maith thú celebrations.
  • # CLIENT SNAPSHOT — 16 named bolded fields (Age, Height, Weight, Experience, Sport, Season phase, Active training status, Current weekly sport load, Sport training days, S&C sessions available, Equipment, Plan duration, Primary goals, Medical notes, Cycle status, Mental performance work), followed by a 2-3 sentence Key context paragraph.
  • # PLAN OVERVIEW — nine required subsections: Structure, Phases, Session split, Standard warm-up (a markdown table prescribed once for use before every session), Scheduling guidance (must use the athlete's reported training days), Programming priorities (numbered 1-5), Injury-prevention emphasis (with cited source), Conditioning rationale, optional Body composition note, optional Mental performance integration.
  • # WEEK-BY-WEEK PLAN — per week: ## Week N: [theme], Focus: line. Per session: Warm-up: line (references the standard table plus 1-2 session-specific primers) → Main session: markdown table (4-6 main lifts only) → blockquote per exercise → Cues:, Cycle consideration:, How to know you are ready to progress:.
  • # COACH NOTES, # CONTRAINDICATED EXERCISES AND SUBSTITUTES, # THINGS TO TRACK — all required at the end.

Mental plan: Same shape with # HOW TO READ THIS PLAN, # ATHLETE SNAPSHOT (12 named fields), # PLAN OVERVIEW (six required subsections including Evidence base with named frameworks), # WEEK-BY-WEEK PLAN (Concept of the week / Daily practice table / Performance moment / Reflection prompt), # ROUTINES TO BUILD (Pre-performance, Refocus, Reset cue), # COACH NOTES (FOR THE CLIENT), # THINGS TO TRACK.

Token caps were bumped to fit the explicit structure: physical 6-week 14k output tokens (was 10k), 12-week 24k (was 18k), with a +2k bonus for specialised tracks. Mental 6-week 12k, 12-week 20k. Configured in programme-finalize.ts at the model call sites; the generator logs a console warning if a plan still hits stop_reason === 'max_tokens'.

7C. Customer Dashboard & Progress Tracking

Once a plan is approved by Emily, the customer logs in to a dashboard and opens their plan as a guided 6 or 12 week journey. Each exercise has a tick box. Sessions auto-complete when every exercise is ticked. Weeks unlock in sequence as the previous week's items are ticked. A stats panel shows percent complete, sessions complete, weeks complete, and day streak. Saoirse the shamrock pops in to celebrate completion. A pilot feedback popup captures a 1-5 rating + free-text comments at the end of every week.

7C.1 Data model

One JSONB column on the Supabase Plans table:

  • Plans.progress (JSONB, default '{}') — keys are "week.session.exercise" strings (e.g. "1.2.3" = week 1, session 2, exercise 3); values are the ISO timestamp the exercise was ticked. For mental plans, the daily-practice table rows are treated as session 1 exercises 1-N (one row per day).
  • Plans.weights (JSONB, default '{}') — same key shape as progress; values are the weight the athlete logged for that exercise (e.g. "65", "BW"). Physical plans only.

7C.1.1 Plan-page rendering pipeline

Each week's markdown content goes through this pipeline in /plan/[id].astro before being injected into the page:

  1. marked.parse — markdown to HTML.
  2. stripEmptyBlocks — drops empty <blockquote> and <p> elements that the markdown sometimes produces between sections.
  3. transformCoachTips — wraps labelled paragraphs (Cycle consideration, Ready to progress, Concept of the week, Reflection prompt) into .coach-tip cards with a Saoirse avatar. The cues label is dropped entirely (covered inline in each exercise description).
  4. dedupePerWeekCoachTips — if the plan generator emitted Cycle/Ready-to-progress per session (old prompt), collapses each to its last occurrence so only one shows per week.
  5. injectExerciseCheckboxes — adds a leading Done column (checkbox per row) and a trailing Weight column (preset-options dropdown) to the exercise table. Mental plans get the Done column only.
  6. mergeExerciseDescriptions — pulls each <blockquote> paragraph (or top-level **Exercise:** paragraph) describing an exercise INTO the matching row's name cell as a lime-green Saoirse banner. Falls back to sentence-level matching when a single paragraph covers multiple exercises (e.g. "Mobility work" describes both 90/90 hip mobility and World's greatest stretch).

7C.2 Customer routes

  • /dashboard — lists every approved plan owned by the signed-in user. Includes the live AI-generated disclaimer banner.
  • /plan/[id] — the single-plan view: stats ring (% complete, sessions complete, weeks complete, day streak), week pills (lock or unlock based on completion), week content (rendered from the plan's markdown), Saoirse coach bubbles on the cues / cycle / progress / concept / reflection labels, per-exercise tick boxes on every table row, celebration modals on session and week completion, pilot feedback popup at end of week.

7C.3 Tick flow

  1. Customer clicks a checkbox on an exercise row in /plan/[id].
  2. Page calls POST /api/plan/toggle-session with { planId, week, session, exercise, completed } and the customer's Supabase access token in Authorization: Bearer ....
  3. Server verifies the token, fetches the Plans row, confirms user_id matches, merges the tick into progress, writes the row back. Returns the updated progress object.
  4. Client updates local state, re-renders the stats ring + week-nav, decides whether to fire the day-complete or week-complete celebration modal.

7C.4 Saoirse celebrations

  • Coach tip bubbles — inside the plan content, paragraphs that start with **Cues:**, **Cycle consideration:**, **How to know you are ready to progress:**, **Concept of the week**, or **Reflection prompt** are post-processed (after marked.parse) into a speech-bubble card with a Saoirse avatar on the left. 7D.3 The race-condition story

Earlier in the pilot, parallel Regenerate clicks (or Regenerate + Recover-intake clicks against the same customer) produced duplicate Plans rows because each call deleted the previous row, reset finalized_at, then independently generated and inserted. The 2-minute duplicate-write guard catches the common human-click race (Plans.created_at >= now - 2 min for the same user_id + plan_type returns 409). A proper atomic lock (DB-level row lock or unique constraint) is on the roadmap for the post-pilot hardening pass.

The 12-month editorial calendar releases content on a fixed schedule (1st August 2026 launch, then monthly). Posts are written and committed ahead of release date, but visitors should only be able to read the ones whose date has arrived. 7B.1 How the lock works

On src/pages/content-hub/training-the-mind.astro and gael-performance-toolkit.astro:

const today = new Date();

  date.toLocaleDateString('en-GB',

7E. Team Plans (Coaches & Clubs)

Coaches apply at /coaches, Emily reviews them at /admin/coach-applications and approves, the coach signs in to /coach-dashboard, generates a squad-wide plan from /coach-intake, the plan goes through the same admin review queue, and once approved the coach gets a private link they share with their squad. No individual player data is collected on this path.

7E.1 Data model

  • coach_applications (Supabase) — one row per public application. status pending / approved / rejected.
  • coaches (Supabase) — one row per approved coach. Email is the unique key; user_id is filled in the first time the coach signs in via Supabase Auth.
  • Plans extension — three new columns: is_team_plan (boolean), share_token (string, unique when set, used as the slug in /team-plan/[token]), coach_id (FK to coaches), and team_intake_data (JSONB of the team intake submitted).

Migration in migrations/004-team-plans-supabase.sql. Run once in the Supabase SQL editor.

7E.2 Flow

  1. Apply. Coach hits /coaches, fills the application form, submits to POST /api/coach-application. Row written to coach_applications with status='pending'; Emily is notified by email.
  2. Approve. Emily reviews at /admin/coach-applications. Approve calls POST /api/admin/approve-coach which marks the application approved, upserts a row into coaches keyed on the lowercase email, and sends the coach a welcome email pointing them at /login.
  3. Sign in. Coach signs in with the same email via Supabase Auth on /login. First time they hit any coach endpoint, getCoachForToken() backfills coaches.user_id from auth.users.id.
  4. Generate. Coach opens /coach-dashboard → /coach-intake, fills the team intake (sport, plan type, duration, season phase, training days, equipment, group size, primary goal). POST /api/coach-intake validates that the caller is an approved coach (lib/coach-auth.ts), generates the plan via generateTeamPlan() in lib/team-plan.ts, and writes a Plans row with is_team_plan=true, a random 32-char share_token, coach_id set, and team_intake_data persisted so the plan can be regenerated later.
  5. Admin review. Team plans appear in the same /admin/plans queue with a lime "Team plan" badge and a green left-border. The share link is visible to Emily so she can preview it.
  6. Approve and share. Emily clicks Approve. Status flips to active; the share link now resolves publicly at /team-plan/[share_token]. The coach sees the live link on their dashboard with a Copy button.
  7. Squad reads. Anyone with the link can read the plan. No login required. The page is read-only (no tick boxes, no progress tracking).

7E.3 Team plan prompt

Lives in src/lib/team-plan.ts. Mirrors the explicit OUTPUT STRUCTURE of the individual prompts but pitched at the squad, not an individual player:

  • Physical team plan: # HOW TO READ THIS PLAN (FOR COACHES) → # TEAM SNAPSHOT (Sport, Squad size, Average experience level, Plan duration, Season phase, Sport training days, Match days, Equipment, Primary team goal) → # PLAN OVERVIEW (Structure, Phases, Session split, Standard warm-up table, Scheduling guidance, Programming priorities, Injury-prevention emphasis, Conditioning rationale, Coach adaptation principles) → # WEEK-BY-WEEK PLAN per session: Warm-up line → Main session table → blockquote per exercise → Cues → Coach adaptation (how to scale up/down for individual players) → Ready-to-progress.
  • Mental team plan: Same shape with # ROUTINES TO BUILD at squad level (pre-match team routine, refocus, reset cue) and a Coach prompt for the squad per week so the coach has a one-line opener for the next training.

Token caps (slightly leaner than individual plans because cycle / medical / mental health fields are absent): physical 6-week 13k, 12-week 22k. Mental 6-week 11k, 12-week 18k.

Stripe has two completely separate environments:

  • Test mode: STRIPE_SECRET_KEY=sk_test_.... Use test card 4242 4242 4242 4242. No real money moves.
  • Live mode: STRIPE_SECRET_KEY=sk_live_.... Real cards, real money.

The environment is controlled by which key is in .env (local) or Vercel environment variables (production).

8.3 Programme-specific success URL

Standard checkouts redirect to /success. Programme purchases redirect to /programme-success?session_id={CHECKOUT_SESSION_ID}&plan=...&token=.... This lets the success page call /api/programme-finalize with the right session ID and intake token.

8.4 Stripe webhook (resilience)

The success page calling /api/programme-finalize from the browser is the happy path. If the customer closes their tab before the page loads, or if a network error breaks the request, the plan would not be generated and the customer would be charged with nothing to show for it. The Stripe webhook closes that gap: Stripe calls our server when a payment completes, regardless of what the browser is doing.

Endpoint: /api/stripe-webhook. Stripe POSTs checkout.session.completed events here. The handler:

  1. Reads the raw request body (required for Stripe signature verification).
  2. Verifies the stripe-signature header against STRIPE_WEBHOOK_SECRET.
  3. Extracts intake_token from session.metadata (set by create-checkout-session.ts when the checkout was created).
  4. Calls finalizeIntake({ token, sessionId }) — the exact same function the browser success-page path uses.

Idempotency

finalizeIntake() is now idempotent: if the intake row already has a finalized_at timestamp, it returns { ok: true, alreadyFinalized: true } without re-generating the plan or re-sending emails. This means both the browser success-page path and the webhook path can fire safely — whichever wins runs the work, and the loser is a no-op.

Setup steps (one-off, before production)

  1. Stripe Dashboard → Developers → Webhooks → Add endpoint.
  2. URL: https://mindthegael.co.uk/api/stripe-webhook
  3. Events to send: checkout.session.completed (and optionally checkout.session.async_payment_succeeded if any payment methods that settle asynchronously are enabled).
  4. Copy the signing secret (starts with whsec_) into the Vercel environment variable STRIPE_WEBHOOK_SECRET.
  5. Redeploy. Stripe's Dashboard has a "Send test webhook" button to confirm the handler returns 200.
Subscription purchases (the £5/£8 blog) also fire checkout.session.completed but carry no intake_token in metadata; the webhook detects this and returns 200 without doing any finalize work. So the same endpoint safely handles both subscription and plan checkouts.

11. Safety Features & Hard Rules

Four hard rules apply across the platform:

11.1 Mental health crisis — hard stop

Both the chatbot and the training-plan flow halt on any mental-health-crisis indicator. The chatbot bypasses the AI and returns a safe message. The training plan stores the intake with a red_flag_id marker but generates no teaser and offers no purchase. Crisis vocabulary is centralised in CRISIS_TERMS (chat.ts) and CRISIS_PATTERNS (red-flags.ts).

11.2 Citation rule

Any claim the AI makes that is NOT from Mind the Gael's own content must be attributable to a real, named source: a peer-reviewed paper (author plus year), a recognised authority (NICE, POGP, Aspetar, NHS, WHO, ACOG, ESHRE), or a well-established named framework. Hallucinated citations are forbidden. The system prompts repeat this rule explicitly.

11.3 No em-dashes

Em-dashes (—) and en-dashes (–) are an AI tell and not in Emily's voice. They are removed from:

  • All user-facing website copy
  • All email bodies
  • All system prompts (so the model doesn't echo them back)
  • All model output (via stripDashes() post-processing in both chat.ts and the plan generators)

Hyphens in compound modifiers ("6-week", "32-year-old") are kept; those are standard English.

11.4 Beginner accessibility

The training-plan system prompts include explicit rules that for beginner-level clients, every S&C term (RPE, sets, reps, deload, progressive overload, tempo) must be defined in plain English the first time it appears.

11.5 Legal touchpoints in code

Three pieces of legal compliance landed in the code in June 2026:

  • Point-of-use intake disclaimer + 18+ checkbox. Both intake modals on src/pages/personal-training.astro render a purple-bordered disclaimer panel directly above the submit button: "AI-generated performance coaching, not medical, psychological, or therapeutic advice", crisis numbers (Samaritans 116 123, Pieta House 1800 247 247), and a required "I confirm I am 18 years of age or older" checkbox. Form submission is blocked without it.
  • Report-this-output link. Every plan email (buildClientPlanEmailHtml in src/lib/email-format.ts) ends with a small "Was anything in this plan unclear, unsafe, or off? Report it here" link routing to /content-feedback?source=plan_email. The query param flags plan-email reports separately from generic content feedback.
  • Cookie consent banner. src/components/CookieBanner.astro renders a bottom-fixed dark green banner with lime border and a "Got it" button. Shows once on first visit, stores consent in localStorage under mtg_cookie_consent, and removes itself from the DOM after acceptance. Included site-wide via Footer.astro. Note: Vercel Analytics is still loaded unconditionally; gating analytics behind consent is a pending follow-up.

15. Glossary

TermDefinition
AstroThe web framework Mind the Gael is built with. Produces static HTML for content pages and runs server endpoints for dynamic functionality.
API endpointA URL on the site that runs server-side code rather than returning a page (e.g. /api/chat processes a chatbot message).
Content collectionAstro's system for managing structured content (markdown files with frontmatter), used here for all blog posts and toolkit articles.
Edge / CDNVercel's global network of servers that serve the site quickly from a location near each visitor.
Environment variableA piece of configuration (like an API key) that lives outside the code, set per environment (local dev, production).
FrontmatterThe YAML block at the top of a markdown file (between the two --- lines) holding metadata like title and publish date.
MarkdownA plain-text formatting syntax for writing content (# for headers, ** for bold, etc.). All blog posts are written in markdown.
Postgres / NeonThe relational database the site uses. Provisioned through Vercel's Storage product; Neon is the underlying provider.
Preview tokenThe UUID stored in intake_sessions when an intake form is submitted. It carries through Stripe and is used by /api/programme-finalize to look up the intake.
ResendThe transactional email service used to send all platform emails.
SaoirseThe name of the chatbot. Greeted with a shamrock mascot of the same name.
Stripe CheckoutStripe's hosted payment page. The site sends a buyer there with a session ID; Stripe handles the rest and redirects them back.
System promptThe text sent to Claude AI at the start of every request, defining who the AI is and how to respond. Roughly equivalent to giving a new employee a brief on their role.
v1 / v2v1 is the launch phase: every training plan goes to Emily for review before being sent to the client. v2 is autonomous: standard plans go straight to the client. Specialised tracks always route to Emily regardless.
VercelThe hosting platform. Pulls code from GitHub, builds the site, serves it globally, runs the API endpoints as serverless functions.
